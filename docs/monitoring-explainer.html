the fine print — exact checks and thresholds

Stack monitors (#187 #249 #268 #304)

One POST to Hasura (dipdup_index + dipdup_head), seven conditions inside the response; any false after 3 retries = Down:

  1. tezos_head.updated_at fresh < N — dead process / hung TzKT websocket;
  2. etherlink_deposit_events fresh < N — stuck evm.events subscription;
  3. etherlink_xtz_deposit_transactions fresh < N — evm.transactions, a separate subscription type that wedges independently;
  4. dipdup_head.timestamp fresh < N — written straight from the realtime channel, bypassing the index queue;
  5. tezos_head behind the websocket head by < G blocks — "running but not keeping up"; threshold 30 used to flap from the full re-match after on_synchronized;
  6. all 4 operation indexes exist ($count = 4) — guards against a broken reindex;
  7. operation indexes behind head by < 600 blocks — normal is 20–148 (9 samples / 25 min across 4 stacks), 600 = 4× margin, detects in ~1.3 h.

Matching checks (#305 #312 #313)

#305 and #312: the 100 freshest unmatched operations, window 1 h – 7 days by block time (l1_transaction.timestamp), not by created_at — indexing time makes all history look "fresh" after a backfill. One hour = matching grace; one week cuts off the permanent residue (~546 historical deposits; withdrawals 0 of 26 796). L1 execution of withdrawals is excluded — it legitimately waits ~2 weeks for cementation (703 pending is normal). Poll 300s × r2.

#313: head − max(inbox.level) < 10 800 blocks (~24 h); messages on prod arrive every ~300 blocks, 28-day max — 8 538. Poll 600s × r3. Interim until a heartbeat key in dipdup_meta written by the loop itself — that will cover the quiet stacks too.

Source probes (#306–#311)

EVM nodes: eth_getBlockByNumber(latest), block fresh ≤ 15 min (#306 60s × r2; #310 #311 120s × r3). Subsquid: GET /height > 0, 300s × r3. TzKT: GET /v1/head ≤ 5 min, 60s × r2. Sonic is the node the indexer actually uses; the older #303 watches evm-mainnet, which the indexer does not read.
